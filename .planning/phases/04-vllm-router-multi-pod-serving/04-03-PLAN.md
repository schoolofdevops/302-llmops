    **Step B — Enable router DEBUG logging to capture backend pod selection:**
    ```bash
    # Get router pod name
    ROUTER_POD=$(kubectl get pods -n llm-serving -l app.kubernetes.io/component=router -o jsonpath='{.items[0].metadata.name}')
    echo "Router pod: $ROUTER_POD"
    ```

    **Step C — Session routing demo (3 requests, same session ID):**
    ```bash
    SESSION_ID="dental-session-001"
    for i in 1 2 3; do
      echo "--- Request $i ---"
      curl -s http://localhost:30201/v1/chat/completions \
        -H "Content-Type: application/json" \
        -H "x-user-id: ${SESSION_ID}" \
        -d "{
          \"model\": \"smollm2-135m-finetuned\",
          \"messages\": [{\"role\": \"user\", \"content\": \"Describe dental hygiene step ${i}\"}],
          \"max_tokens\": 30
        }" | python3 -m json.tool | grep -E "content|choices"
      sleep 2
    done
    kill $LOG_PID 2>/dev/null || true
    ```
    In the router logs, look for lines referencing the backend pod IP. All 3 requests should route to the same backend IP because the session key `dental-session-001` hashes to the same backend. If the log shows different IPs for the same session ID, record this as a deviation in the summary (may indicate session routing is not working as expected on CPU).

    **Step D — Send a request with a DIFFERENT session ID to confirm different backend assignment:**
    ```bash
    ROUTER_POD=$(kubectl get pods -n llm-serving -l app.kubernetes.io/component=router -o jsonpath='{.items[0].metadata.name}')
    kubectl logs -n llm-serving "$ROUTER_POD" --tail=0 -f &
    LOG_PID=$!

    curl -s http://localhost:30201/v1/chat/completions \
      -H "Content-Type: application/json" \
      -H "x-user-id: dental-session-002" \
      -d '{"model":"smollm2-135m-finetuned","messages":[{"role":"user","content":"What causes cavities?"}],"max_tokens":30}' \
      | python3 -m json.tool | grep "content"

    sleep 3
    kill $LOG_PID 2>/dev/null || true
    ```
    Capture log output showing backend IP assignment for session-002. Compare with session-001 assignment — they should differ (or may be the same if only 1 backend is reachable; document either outcome).

    **Step E — Verify vllm:num_requests_waiting metric exported by CPU vLLM image:**
    ```bash
    # Pick one backend pod
    BACKEND_POD=$(kubectl get pods -n llm-serving -l app.kubernetes.io/component=vllm -o jsonpath='{.items[0].metadata.name}')
    kubectl exec -n llm-serving "$BACKEND_POD" -- wget -qO- http://localhost:8000/metrics \
      | grep "vllm:num_requests_waiting"
    ```

    Expected output: one or more lines like `vllm:num_requests_waiting{...} 0.0`. If the metric is absent, the KEDA scaler will fail silently. Record metric presence or absence in the summary — this is RESEARCH.md Assumption A5.

    ```bash
      wget -qO- "http://localhost:9090/api/v1/query?query=vllm%3Anum_requests_waiting" \
      | python3 -m json.tool | grep -E "result|metric|value"
    ```

<task type="auto">
  <name>Task 2: KEDA scale-up demo under load</name>
  <files></files>
  <action>
    This task drives load to trigger KEDA scale-up. The threshold is `sum(vllm:num_requests_waiting) > 5`. With CPU vLLM at `maxNumSeqs: 1`, a concurrent burst of requests quickly fills the waiting queue.

    **Step A — Resolve actual backend Deployment name + record baseline replica count:**
    ```bash
    # Resolve actual Deployment name from the ScaledObject (may differ from chart convention)
    BACKEND_DEPLOY=$(kubectl get scaledobject -n llm-serving -o jsonpath='{.items[0].spec.scaleTargetRef.name}')
    echo "Backend Deployment: $BACKEND_DEPLOY"

    # Record baseline replica count
    kubectl get deployment -n llm-serving "$BACKEND_DEPLOY" -o jsonpath='{.spec.replicas}'
    echo " replicas (baseline)"
    ```

    **Step B — Run load burst using `hey` (already in COURSE_VERSIONS.md as `williamyeh/hey:latest`):**
    ```bash
    # Run hey load generator as a K8s Job so it hits the cluster-internal service
    kubectl run hey-load --rm --restart=Never --image=williamyeh/hey:latest \
      --namespace llm-serving \
      -- /hey -n 20 -c 5 -t 60 \
      -m POST \
      -H "Content-Type: application/json" \
      -d '{"model":"smollm2-135m-finetuned","messages":[{"role":"user","content":"Explain dental care"}],"max_tokens":50}' \
      http://vllm-stack-router-service.llm-serving.svc.cluster.local/v1/chat/completions
    ```
    Note: `hey-load` pod name must not conflict with existing pods; `--rm` removes it after completion.

    Alternatively, if the hey Job approach is blocked (RBAC or image pull issues), run a simpler curl loop from the host:
    ```bash
    for i in $(seq 1 15); do
      curl -s http://localhost:30201/v1/chat/completions \
        -H "Content-Type: application/json" \
        -d '{"model":"smollm2-135m-finetuned","messages":[{"role":"user","content":"Explain dental care"}],"max_tokens":50}' &
    done
    wait
    ```
    The background `&` sends all 15 requests concurrently, driving the waiting queue above the threshold of 5.

    **Step C — Observe KEDA response:**
    ```bash
    # Use the actual Deployment name resolved in Step A (not hardcoded)
    BACKEND_DEPLOY=$(kubectl get scaledobject -n llm-serving -o jsonpath='{.items[0].spec.scaleTargetRef.name}')
    # Poll replica count every 15 seconds for up to 5 minutes
    for i in $(seq 1 20); do
      echo "$(date +%H:%M:%S) replicas: $(kubectl get deployment -n llm-serving "$BACKEND_DEPLOY" -o jsonpath='{.status.readyReplicas}' 2>/dev/null || echo '0')"
      sleep 15
    done
    ```

    **Step D — Record KEDA events:**
    ```bash
    kubectl describe scaledobject vllm-stack-smollm2-scaledobject -n llm-serving | grep -A 20 "Events:"
    kubectl get events -n llm-serving --field-selector reason=SuccessfulRescale
    ```

    **Step E — Cool down:**
    Wait for requests to finish. KEDA `cooldownPeriod: 360` means replicas will scale back to `minReplicaCount: 1` after 360 seconds of idle. Record the time-to-scale-up (when replicas went from 1 to 2+) in the summary. Do NOT wait for scale-down before proceeding — the cooldown is 6 minutes and the demo evidence is already captured.

    Document in summary:
    - Whether scale-up was observed (replicas changed from 1 to 2+)
    - If NOT observed: likely reasons (metric not exported; Prometheus not scraping; KEDA threshold not exceeded; note RESEARCH.md Assumption A5)
    - Timestamp of scale-up event if observed
  </action>
