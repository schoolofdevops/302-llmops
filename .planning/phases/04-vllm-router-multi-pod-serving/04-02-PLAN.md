    ```bash
    docker pull lmcache/lmstack-router:v0.1.11
    docker inspect lmcache/lmstack-router:v0.1.11 --format '{{.Architecture}}'
    ```
    If the output is `amd64` (expected on all platforms), proceed. If running on an arm64 Mac and Docker Desktop Rosetta is NOT enabled, STOP and add the following note to the task output before proceeding: "ARM64 GATE: Docker Desktop Rosetta emulation must be enabled under Settings > Features in Development before helm install. This plan records the gate but cannot automate the setting." On Intel Mac or Linux amd64, this gate is a no-op.

    **Step C — helm template dry-run to verify emptyDir rendering:**
    ```bash
    helm template vllm-stack vllm/vllm-stack \
      --version 0.1.11 \
      -f course-code/labs/lab-07/solution/k8s/00-values-vllm-router.yaml \
      -n llm-serving \
      | grep -A 30 "initContainers"
    ```

    **Evaluate the output:**

    CASE 1 — If `initContainers:` block appears with `model-download` and `/model` volumeMount: The chart renders the initContainer correctly. Proceed to helm install.

    CASE 2 — If `initContainers:` block is absent or emptyDir volume is not wired to the backend pod spec: The chart does NOT support emptyDir-based model loading through values. Apply the FALLBACK path:
    - The solution values file must be updated: remove `modelSpec[].initContainer` and add two standalone Deployment + Service resources for the backends (mirroring the Phase 03 pattern from `course-code/labs/lab-06/solution/k8s/30-deploy-vllm-disk.yaml` but with names `vllm-smollm2-backend-0` and `vllm-smollm2-backend-1`).
    - Change `routerSpec` to add `serviceDiscovery: static` and list `staticBackends:` pointing at the two backend Service ClusterIPs.
    - Document the deviation: add a comment block at the top of the solution values file stating: "FALLBACK MODE: chart 0.1.11 emptyDir not supported via values; using static backend discovery."
    Record which case applied in the plan execution summary.

    **Step D — helm install:**
    ```bash
    helm install vllm-stack vllm/vllm-stack \
      --version 0.1.11 \
      --namespace llm-serving \
      --create-namespace \
      -f course-code/labs/lab-07/solution/k8s/00-values-vllm-router.yaml \
      --wait \
      --timeout 600s
    ```
    The `--wait` flag blocks until all pods are Ready or timeout. With CPU model download + startup, 600s may be tight; if it times out, check pod status and wait for manual readiness before verifying.

    **Step E — Verify router + backends:**
    ```bash

    kubectl get pods -n llm-serving -l app.kubernetes.io/component=router

    kubectl get pods -n llm-serving -l app.kubernetes.io/component=vllm

    kubectl get endpoints -n llm-serving
    ```

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| host → NodePort 30201 | Unauthenticated HTTP to lmstack-router; acceptable for isolated lab |
| lmstack-router → backends | Pod-IP based discovery inside cluster; no mTLS in lab |
| initContainer → MinIO | HTTP (not HTTPS) to MinIO at minio.minio:9000; credentials hardcoded in values (minio/minio123) |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-04-03 | Spoofing | `x-user-id` session key header | accept | Client-controlled header is pedagogically intentional; lab explicitly notes this is NOT auth (per RESEARCH.md security domain) |
| T-04-04 | Information Disclosure | MinIO credentials in values.yaml | accept | Hardcoded lab credentials (minio/minio123) are in version-controlled course files; acceptable for isolated KIND lab; warn students not to use in production |
| T-04-05 | Elevation of Privilege | requestGPU: 0 misconfiguration | mitigate | Values file explicitly sets `requestGPU: 0`; RESEARCH.md Pitfall documents that omitting this causes indefinite Pending; verified via helm template dry-run |
| T-04-SC | Tampering | helm install pulls lmstack-router:v0.1.11 | mitigate | Package Legitimacy Audit in RESEARCH.md approves this image (official LMCache org, Docker Hub); pinned to `v0.1.11` (not latest) |
</threat_model>
