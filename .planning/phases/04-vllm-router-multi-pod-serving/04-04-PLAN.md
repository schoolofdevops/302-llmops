    **1. Lab Overview**
    - Introduce vLLM Production Stack: what the router does, why session routing matters (KV-cache affinity), why KEDA scales backends not router.
    - Architecture diagram as a plain-text ASCII box diagram (copy from RESEARCH.md with minimal adaptation).
    - Prerequisites callout: Pattern A and Pattern B must be scaled to 0 before starting.
    - arm64 Mac callout: warn students to enable Docker Desktop Rosetta before proceeding if on Apple Silicon (Settings > Features in Development > Use Rosetta for x86_64/amd64 emulation). Include the platform check command: `docker pull lmcache/lmstack-router:v0.1.11 && docker inspect lmcache/lmstack-router:v0.1.11 --format '{{.Architecture}}'`.

    **2. Memory Prerequisites**
    ```bash
    # Scale down other serving patterns to free memory (required — two 4Gi backends need headroom)
    kubectl scale deployment vllm-smollm2 -n llm-serving --replicas=0
    kubectl scale deployment vllm-smollm2-disk -n llm-serving --replicas=0
    kubectl get deployment -n llm-serving   # Verify both show 0/0 READY
    ```

    **3. Add Helm Repo**
    ```bash
    helm repo add vllm https://vllm-project.github.io/production-stack
    helm repo update
    helm search repo vllm/vllm-stack --versions | head -5
    ```

    **4. Values File**
    Show the STARTER values file (`course-code/labs/lab-07/starter/k8s/00-values-vllm-router.yaml`) with guidance on filling in the TODO blanks. For each TODO, give a hint (e.g., "check kind-config.yaml for the router NodePort"). Show the complete solution values file after, in a collapsible `<details>` block labeled "Solution: Complete values".

    Explain each key decision in the values:
    - `requestGPU: 0` — mandatory for CPU KIND cluster; omitting causes indefinite Pending
    - `routingLogic: session` — enables KV-cache affinity; requires `x-user-id` header; prefixaware is not used (requires LMCache controller)
    - `nodePort: 30201` — requires vllm-stack 0.1.11; not present in 0.1.10
    - Prometheus serverAddress — must match kps release name: `kps-kube-prometheus-stack-prometheus.monitoring.svc.cluster.local:9090`

    **5. helm template dry-run** (Pitfall prevention)
    ```bash
    helm template vllm-stack vllm/vllm-stack \
      --version 0.1.11 \
      -f course-code/labs/lab-07/solution/k8s/00-values-vllm-router.yaml \
      -n llm-serving \
      | grep -A 20 "initContainers"
    ```
    Add a callout: "If the initContainers block does not appear, see the Troubleshooting section at the bottom of this lab."

    **6. Helm Install**
    ```bash
    helm install vllm-stack vllm/vllm-stack \
      --version 0.1.11 \
      --namespace llm-serving \
      --create-namespace \
      -f course-code/labs/lab-07/solution/k8s/00-values-vllm-router.yaml \
      --wait \
      --timeout 600s
    ```
    Explain that `--wait` blocks until pods are Ready. CPU model download + startup takes 3-8 minutes.

    **7. Verify Deployment**
    ```bash
    # Router pod
    kubectl get pods -n llm-serving -l app.kubernetes.io/component=router
    # Backend pods (2 expected)
    kubectl get pods -n llm-serving -l app.kubernetes.io/component=vllm
    # Endpoints showing 2 backend IPs
    kubectl get endpoints -n llm-serving
    # Router health check
    curl -s http://localhost:30201/health
    ```

    **8. Test: Router vs Plain vLLM (Pattern A comparison)**
    ```bash
    # Test router at 30201
    curl -s http://localhost:30201/v1/chat/completions \
      -H "Content-Type: application/json" \
      -d '{"model":"smollm2-135m-finetuned","messages":[{"role":"user","content":"What is plaque?"}],"max_tokens":50}' \
      | python3 -m json.tool
    # Router is transparent — same response format as plain vLLM
    ```

    **9. Session Routing Demo**
    Show the 3-request session demo with x-user-id header. Explain that session-001 always routes to the same backend because the router hashes the session key to a consistent backend selection. Include the router log tailing command so students can see backend IP assignment.

    **10. KEDA ScaledObject**
    ```bash
    kubectl get scaledobject -n llm-serving
    kubectl describe scaledobject -n llm-serving
    ```
    Explain what students are reading: READY, ACTIVE, scaleTargetRef (must be the backend Deployment, NOT the router), trigger type prometheus, metric name with colon notation (`vllm:num_requests_waiting`).

    **11. Load Test + Scale-Up Observation**
    Show the curl loop load test and replica count polling. Note cooldown period is 360s. Include `kubectl get events -n llm-serving --field-selector reason=SuccessfulRescale` to capture scale-up events.

    **12. Resource Budget** (insert actual numbers from 04-03-SUMMARY.md budget table)

    **13. Teardown** (CRITICAL — must restore cluster for Phase 05)
    ```bash
    # Remove vllm-stack
    helm uninstall vllm-stack -n llm-serving

    # Verify Phase 04 pods are gone
    kubectl get pods -n llm-serving

    # Restore Pattern A for Phase 05 baseline
    kubectl scale deployment vllm-smollm2 -n llm-serving --replicas=1
    kubectl rollout status deployment/vllm-smollm2 -n llm-serving --timeout=600s

    # Verify headroom restored
    kubectl top nodes
    ```
    Add a warning callout: "Do NOT skip teardown before Phase 05. Running all three serving patterns simultaneously exceeds the 16GB RAM budget."

    **14. Troubleshooting** section covering:
    - Router pod `exec format error` → Enable Rosetta (arm64 Mac)
    - KEDA ScaledObject `READY: False` with Prometheus connection error → wrong `serverAddress`; run `kubectl get svc -n monitoring -l app.kubernetes.io/name=prometheus -o name` to get correct name
    - Backend pods Pending → likely `requestGPU` not set to 0; check `kubectl describe pod`
    - initContainers not in helm template output → emptyDir not supported via values; see fallback in Pitfall 5 of RESEARCH.md (static backend mode)
    - `vllm:num_requests_waiting` metric absent → KEDA will be inactive; verify `kubectl exec ... wget -qO- http://localhost:8000/metrics | grep vllm`

    **Lab summary** section listing what was learned: router deployment pattern, session routing, KEDA Prometheus-driven scaling, memory discipline between serving patterns.
