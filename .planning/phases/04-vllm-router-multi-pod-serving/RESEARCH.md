# Phase 04: vLLM Router Multi-Pod Serving - Research

**Researched:** 2026-06-15
**Domain:** vLLM Production Stack (Helm chart), lmstack-router, KEDA Prometheus scaler, KIND arm64 compatibility
**Confidence:** HIGH (Helm chart values and templates verified directly from the official repo; Docker Hub tags verified; KEDA template verified from source)

## Summary

Phase 04 deploys the vLLM Production Stack router (`lmcache/lmstack-router`) fronting two CPU vLLM backend pods using the `vllm-stack` Helm chart. The core technical challenges are: (1) the chart's GPU-first defaults must be explicitly overridden for CPU-only KIND backends; (2) the `lmstack-router` image is **amd64-only** — no arm64 variant exists, which is the single biggest risk for Apple Silicon Mac instructors/students; (3) the roadmap targets chart version 0.1.10 but 0.1.11 adds a `routerSpec.nodePort` value that eliminates a post-install patch step; and (4) the KEDA `ScaledObject` is rendered by the chart itself (not hand-rolled) when `keda.enabled: true` is set in `modelSpec`.

The vllm-stack 0.1.11 chart is available on the official Helm repo (`https://vllm-project.github.io/production-stack`). Chart 0.1.10 is also available but does **not** have `routerSpec.nodePort` support in its service template — you would need a post-install Service patch to expose NodePort 30201. Recommendation: **bump to 0.1.11** to get NodePort-in-values support and `prefixaware` routing option.

**Primary recommendation:** Use `vllm-stack` chart 0.1.11 (not 0.1.10 as originally scoped) to get native NodePort config; pin router image to `lmcache/lmstack-router:v0.1.11`; use `routingLogic: "session"` (safest for CPU lab — no LMCache controller dependency); set `requestGPU: 0` and `runtimeClassName: ""` in `servingEngineSpec`; inject CPU env vars via `modelSpec[].env`.

### Pattern 1: CPU Backend via vllm-stack Helm values

**What:** Override the GPU-first defaults to deploy CPU-only vLLM serving engines.
**When to use:** Any CPU-only KIND cluster deployment of vllm-stack.

```yaml
# Source: /tmp/vllm-stack/values.yaml (extracted from vllm-stack-0.1.10.tgz)
# and verified against 0.1.11 templates

servingEngineSpec:
  runtimeClassName: ""          # Empty string = no GPU runtime class
  modelSpec:
  - name: "smollm2"
    repository: "schoolofdevops/vllm-cpu-nonuma"
    tag: "0.9.1"
    modelURL: "/model"          # Local path — model pre-loaded via initContainer
    replicaCount: 2

    requestCPU: 4
    requestMemory: "4Gi"
    requestGPU: 0               # Zero GPU → no nvidia.com/gpu resource requested

    vllmConfig:
      maxModelLen: 4096
      dtype: "float32"
      maxNumSeqs: 1
      extraArgs:
        - "--disable-frontend-multiprocessing"
        - "--served-model-name=smollm2-135m-finetuned"

    env:
    - name: VLLM_TARGET_DEVICE
      value: "cpu"
    - name: VLLM_CPU_KVCACHE_SPACE
      value: "2"
    - name: OMP_NUM_THREADS
      value: "4"
    - name: VLLM_CPU_OMP_THREADS_BIND
      value: "auto"

    keda:
      enabled: true
      minReplicaCount: 1
      maxReplicaCount: 3
      pollingInterval: 15
      cooldownPeriod: 360
      triggers:
        - type: prometheus
          metadata:
            serverAddress: "http://kps-kube-prometheus-stack-prometheus.monitoring.svc.cluster.local:9090"
            metricName: "vllm:num_requests_waiting"
            query: 'sum(vllm:num_requests_waiting{model="smollm2"})'
            threshold: "5"

routerSpec:
  repository: "lmcache/lmstack-router"
  tag: "v0.1.11"
  imagePullPolicy: "IfNotPresent"
  replicaCount: 1
  routingLogic: "session"
  sessionKey: "x-user-id"
  serviceType: NodePort
  nodePort: 30201              # Available in 0.1.11 chart only
  servicePort: 80
  resources:
    requests:
      cpu: 400m
      memory: 1000Mi
    limits:
      memory: 1000Mi
```

### Pattern 2: Model Path — initContainer + emptyDir (same as Phase 03)

**What:** The vllm-stack `modelSpec[].initContainer` field injects a model-download initContainer.
**When to use:** Loading model from MinIO into emptyDir before the vLLM backend starts.

```yaml
# Source: /tmp/vllm-stack/values.yaml modelSpec comment block
initContainer:
  name: model-download
  image: "quay.io/minio/mc:RELEASE.2024-11-21T17-21-54Z"
  command:
    - sh
    - -c
    - |
      mc alias set minio http://minio.minio:9000 minio minio123
      mc cp --recursive minio/models/smollm2-finetuned/ /model/
      touch /model/READY
  resources:
    requests:
      cpu: 200m
      memory: 128Mi
      ephemeral-storage: 1Gi
    limits:
      cpu: 500m
      memory: 256Mi
      ephemeral-storage: 2Gi

extraVolumes:
  - name: model
    emptyDir:
      sizeLimit: 1Gi
extraVolumeMounts:
  - name: model
    mountPath: /model
```

> NOTE: The initContainer approach in the Helm chart differs slightly from Phase 03's standalone Deployment. The modelURL must still point to `/model` for the vLLM server command. Verify that `extraVolumes`/`extraVolumeMounts` keys exist in the 0.1.11 schema — if not, the emptyDir must be added via `extraVolumes` in `servingEngineSpec`.

### Pattern 3: NodePort exposure workaround for chart 0.1.10

**What:** If pinned to 0.1.10 chart (no `routerSpec.nodePort` in values), patch the Service after install.
**When to use:** Only if 0.1.10 is required for course version consistency.

```bash
# Post-install Service patch for 0.1.10
kubectl patch svc vllm-stack-router-service -n llm-serving \
  -p '{"spec":{"type":"NodePort","ports":[{"port":80,"targetPort":8000,"nodePort":30201,"name":"router-sport"}]}}'
```

### Anti-Patterns to Avoid

- **Using `latest` tag for router image:** `imagePullPolicy: Always` + `latest` means different bytes every pod restart. Pin to `v0.1.11`.
- **Setting `requestGPU: 1` on a CPU KIND cluster:** The pod will stay `Pending` indefinitely (no GPU resource). Always set `requestGPU: 0`.
- **Leaving `runtimeClassName: "nvidia"` (the hidden default):** The chart's `_helpers.tpl` renders `runtimeClassName` only when the value is non-empty — but the values comment says "e.g. nvidia". Setting `runtimeClassName: ""` explicitly is safe and required.
- **Using `prefixaware` routing without LMCache:** Prefix-aware routing in 0.1.11 requires `lmcacheConfig.enabled: true` and a running LMCache controller. On CPU/KIND without LMCache, use `session` routing instead.
- **Targeting KEDA at the router pod:** KEDA scales backends (the Deployment named `{release}-{modelname}-deployment-vllm`), not the router. The router is a single replica.
- **Running Phase 04 backends alongside Pattern A/B:** Two CPU vLLM backends (4Gi each) + Pattern A (4Gi) + Pattern B (4Gi) = ~16Gi, which exceeds the 9.7GiB Docker Desktop limit. Scale down Pattern A and B before Phase 04.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Session-affinity routing to vLLM pods | Custom nginx/HAProxy config | `lmstack-router` + `routingLogic: session` | Router already implements session-key extraction and pod affinity; nginx doesn't understand vLLM model semantics |
| KEDA ScaledObject for vLLM backends | Manual `ScaledObject` YAML | `modelSpec[].keda.enabled: true` in Helm values | Chart generates the ScaledObject with correct deployment name convention `{release}-{model}-deployment-vllm`; hand-rolled names will drift |
| Prometheus metric aggregation for multiple pods | Custom recording rule | `sum(vllm:num_requests_waiting{model="smollm2"})` in KEDA trigger query | PromQL already supports cross-pod aggregation |
| Backend health checks | Custom probe script | vLLM's built-in `/health` endpoint (already in chart probes) | The chart's `startupProbe`, `readinessProbe`, `livenessProbe` target `/health`; `failureThreshold: 60` handles slow CPU start |

**Key insight:** The vllm-stack Helm chart was designed to handle the KEDA + Prometheus + router + backend wiring as a unit. Deconstructing it into separate YAMLs loses the naming convention guarantees that let KEDA find the right Deployment.

## Common Pitfalls

### Pitfall 1: arm64/Apple Silicon — lmstack-router is amd64-only

**What goes wrong:** `kubectl get pods` shows `lmstack-router` pod in `ImagePullBackOff` or `CrashLoopBackOff` with `exec format error`. On macOS Apple Silicon (arm64), the amd64 image cannot run under Docker Desktop without Rosetta emulation.

**Why it happens:** Every `lmcache/lmstack-router` tag on Docker Hub is `linux/amd64` only — confirmed by checking all tags via Docker Hub API. No `linux/arm64` manifest exists as of 2026-06-15.

**How to avoid:**
- Option A: Enable "Use Rosetta for x86_64/amd64 emulation on Apple Silicon" in Docker Desktop > Settings > Features in Development. KIND on Mac will then pull and run the amd64 image via Rosetta. Performance will be lower but functional.
- Option B: Run Phase 04 on a Linux amd64 machine (CI, cloud VM, or x86 laptop).
- Option C: Build a custom arm64 router image from `github.com/vllm-project/production-stack` source and push to `kind-registry:5001/lmstack-router:v0.1.11-arm64`.

**Warning signs:** `exec format error` in pod logs; `ImagePullBackOff` with `no match for platform in manifest` error message (seen on Docker 29 containerd-mode KIND — see kubernetes-sigs/kind issue #4050).

**Lab note:** The lab guide should include a platform gate check step:
```bash
docker inspect lmcache/lmstack-router:v0.1.11 --format '{{.Architecture}}'
# Expected: amd64  (if arm64, Rosetta must be enabled)
```

### Pitfall 2: vllm-stack 0.1.10 vs 0.1.11 NodePort incompatibility

**What goes wrong:** With chart 0.1.10, setting `routerSpec.serviceType: NodePort` exposes a random port (30000-32767), not port 30201. `kind-config.yaml` doesn't have a random port mapped.

**Why it happens:** The `routerSpec.nodePort` value was added in chart 0.1.11. The 0.1.10 `service-router.yaml` template does not render a `nodePort:` field even when `serviceType: NodePort`.

**How to avoid:** Use chart 0.1.11 (recommended) OR post-patch the service after chart 0.1.10 install:
```bash
kubectl patch svc vllm-stack-router-service -n llm-serving \
  -p '{"spec":{"type":"NodePort","ports":[{"port":80,"targetPort":8000,"nodePort":30201,"name":"router-sport"},{"port":9000,"targetPort":9000,"name":"lmcache-port"}]}}'
```

### Pitfall 3: KEDA deployment name mismatch

**What goes wrong:** Manual `ScaledObject` targets wrong Deployment name; KEDA cannot scale backends.

**Why it happens:** The chart generates the backend Deployment as `{helm-release-name}-{modelSpec.name}-deployment-vllm`. If the release is `vllm-stack` and model name is `smollm2`, the Deployment is `vllm-stack-smollm2-deployment-vllm`. A hand-rolled ScaledObject targeting `vllm-backend` or `vllm-smollm2` will fail silently.

**How to avoid:** Use `keda.enabled: true` in `modelSpec` values and let the chart generate the ScaledObject. Verify after install:
```bash
kubectl get scaledobject -n llm-serving
kubectl get deployment -n llm-serving
# Names must align: vllm-stack-smollm2-deployment-vllm
```

### Pitfall 4: Prometheus server address mismatch

**What goes wrong:** KEDA ScaledObject shows `READY: False` / `Active: Unknown`; `kubectl describe scaledobject` shows Prometheus connection error.

**Why it happens:** The chart default `serverAddress: "http://prometheus-operated.monitoring.svc:9090"` assumes a standard kube-prometheus-stack install with release name `prometheus`. This project uses release name `kps` — so the correct address is `kps-kube-prometheus-stack-prometheus.monitoring.svc.cluster.local:9090`.

**How to avoid:** Always override `serverAddress` in the KEDA trigger. As noted in COURSE_VERSIONS.md: "KEDA Prometheus trigger uses Service name `kps-kube-prometheus-stack-prometheus.monitoring.svc.cluster.local:9090`."

```bash
# Verify the correct Prometheus Service name before writing values
kubectl get svc -n monitoring -l app.kubernetes.io/name=prometheus -o name
```

### Pitfall 5: CPU vLLM model path with Helm chart initContainer

**What goes wrong:** vLLM backend pod starts but logs `FileNotFoundError` for model files; pod crashloops.

**Why it happens:** The Helm chart's `deployment-vllm-multi.yaml` template builds the vLLM serve command using `$modelSpec.modelURL` as the model argument. When `modelURL: "/model"` is set (local path), the container's command becomes `vllm serve /model --host ...`. But if `extraVolumes`/`extraVolumeMounts` aren't wired through the template, the `/model` path is empty.

**How to avoid:** Check whether `servingEngineSpec.extraVolumes` / `servingEngineSpec.extraVolumeMounts` are supported in the 0.1.11 chart values schema. If not, the alternative is to use the `initContainer` + `pvcStorage` path OR to keep the Phase 03 disk deployment running and point the router at it (static service discovery mode). This is the **primary open question** for Phase 04 planning — see Open Questions.

### Pitfall 6: vLLM metrics metric name — colon vs underscore in KEDA query

**What goes wrong:** KEDA cannot find the metric `vllm:num_requests_waiting` in Prometheus; ScaledObject stays `Active: Unknown`.

**Why it happens:** When vLLM exports the metric, it uses the name `vllm:num_requests_waiting` (with colon) in the `/metrics` Prometheus text format. When Prometheus scrapes it, the colon is preserved in the metric name (unlike Prometheus recording rules which convert colons). In PromQL queries, you query `vllm:num_requests_waiting` (with colon) — NOT `vllm_num_requests_waiting` (underscore). This is confirmed by the chart template which uses `query: vllm:num_requests_waiting{model="{{ $modelSpec.name }}"}` with colon notation.

**How to avoid:** Always use the colon form in KEDA queries: `vllm:num_requests_waiting`. Verify before deploying:
```bash
kubectl exec -n monitoring deploy/kps-kube-prometheus-stack-prometheus -- \
  wget -qO- "http://localhost:9090/api/v1/query?query=vllm:num_requests_waiting" | python3 -m json.tool
```

### Pitfall 7: `--disable-frontend-multiprocessing` required for CPU backend via Helm chart command

**What goes wrong:** vLLM CPU backend pod crashes with multiprocessing-related OOM errors or hangs indefinitely.

**Why it happens:** The Helm chart renders `vllm serve {modelURL} --host 0.0.0.0 --port 8000 ...` using `extraArgs` for additional flags. The `--disable-frontend-multiprocessing` flag (required for CPU mode) must be passed via `vllmConfig.extraArgs`.

**How to avoid:** Include in `modelSpec[].vllmConfig.extraArgs`:
```yaml
vllmConfig:
  extraArgs:
    - "--disable-frontend-multiprocessing"
    - "--served-model-name=smollm2-135m-finetuned"
```

## Open Questions

1. **Can the Helm chart's `servingEngineSpec.modelSpec[].initContainer` + emptyDir mount work with `modelURL: "/model"`?**
   - What we know: The chart template renders `vllm serve {modelURL}` as the command. The `initContainer` field and `extraVolumes`/`extraVolumeMounts` exist in the values comments.
   - What's unclear: Whether `extraVolumes`/`extraVolumeMounts` at the `servingEngineSpec` level are templated into the backend Deployment's pod spec; and whether vLLM 0.9.1 accepts a local directory path in the `serve {modelURL}` position.
   - Recommendation: **The plan should include a Wave 0 task to `helm template` the chart and inspect the rendered Deployment YAML before committing to this approach.** Alternative: deploy 2 standalone Deployment/Service resources (as in Phase 03) and point the router at them via `staticBackends` + `serviceDiscovery: static`.

2. **Should the course pin to chart 0.1.10 (as stated in ROADMAP) or upgrade to 0.1.11?**
   - What we know: 0.1.10 is missing `routerSpec.nodePort` in values; 0.1.11 adds it. 0.1.11 also adds `prefixaware` routing.
   - What's unclear: Whether the ROADMAP's "0.1.10" pin was intentional (course versioning decision) or a draft placeholder before 0.1.11 was available.
   - Recommendation: **Use 0.1.11** (released 2026-05-07, available in Helm repo). Update COURSE_VERSIONS.md and ROADMAP's Phase 04 description accordingly.

3. **Does `lmstack-router` run correctly on Apple Silicon with Docker Desktop Rosetta emulation?**
   - What we know: The image is amd64-only; Rosetta 2 in Docker Desktop can emulate amd64.
   - What's unclear: Whether any Rosetta-incompatible instruction sets (AVX-512, etc.) are used in the router binary.
   - Recommendation: **Live-cluster verification gate** — test on a Mac M-series before including in the course. If it fails, the fallback is an instructor-provided arm64 build or a course note directing Mac arm64 students to use Rosetta.
