## Part D — Apply the ScaledObject and the HPA

The ScaledObject is the headline:

```bash
kubectl apply -f k8s/80-keda-scaledobject-vllm.yaml
```

What it says (annotated):
```yaml
spec:
  scaleTargetRef:
    name: vllm-smollm2          # The Deployment we want to scale
  pollingInterval: 15           # KEDA queries Prometheus every 15s
  cooldownPeriod: 300           # After triggers go quiet, wait 5 min before scaling down
  minReplicaCount: 1            # NEVER scale to 0 — vLLM cold start is 60-180s on CPU
  maxReplicaCount: 3            # Bound by 12-16 GB Docker Desktop budget (D-21)
  triggers:
  - type: prometheus
    metadata:
      serverAddress: http://<RESOLVED_PROM_SVC>.monitoring.svc.cluster.local:9090
      threshold: '1'             # Sustained ≥1 requests waiting → add a replica
      query: vllm:num_requests_waiting
```

:::tip Why queue depth, not CPU?
vLLM keeps CPU pegged near 100% even when idle (background work, KV-cache management). HPA on CPU would either (a) keep adding replicas forever or (b) thrash. The right saturation signal for an LLM serving engine is the request queue: when requests pile up faster than the engine can finish them, you need more replicas. KEDA's Prometheus scaler reads exactly that.
:::

Wait 30s, then confirm KEDA accepted the ScaledObject:
```bash
kubectl get scaledobject vllm-smollm2 -n llm-serving
# NAME           SCALETARGETKIND   ...   READY   ACTIVE
# vllm-smollm2   apps/v1.Deployment   ... True    False
```

`READY=True` means KEDA connected to Prometheus and the query parsed. `ACTIVE=False` is correct right now — there's no load.

Apply the contrast HPA:
```bash
kubectl apply -f k8s/80-hpa-rag-retriever.yaml
```

After ~30s metrics-server has scraped:
```bash
kubectl get hpa rag-retriever -n llm-app
# NAME             REFERENCE                 TARGETS         MINPODS   MAXPODS   REPLICAS
# rag-retriever    Deployment/rag-retriever  <baseline>%/60%   1         2         1
```

The `<baseline>%/60%` line is HPA reading the actual CPU usage off metrics-server. If `TARGETS` shows `<unknown>/60%`, metrics-server hasn't scraped yet — wait another 30s.

## Part F — Run the loadgen and watch KEDA react (D-04 demo win)

Open Grafana split-screen alongside your terminal — that's the demo. Then:

```bash
bash scripts/run-loadgen.sh
```

The script applies a `Job` that runs `williamyeh/hey:latest` for **180 seconds** at **8 RPS** (4 concurrent workers × 2 RPS each), POSTing the canonical body:
```json
{"model":"smollm2-135m-finetuned","prompt":"What treatments does Smile Dental offer?","max_tokens":32}
```

:::warning Pitfall — model name
The `model` field MUST be `smollm2-135m-finetuned` (the value of `--served-model-name` from `lab-04/solution/k8s/30-deploy-vllm.yaml`). `smollm2` returns 404. The Job manifest already has the right value.
:::

While the loadgen runs the script tails replica count + ScaledObject Active status every 15s. Watch the Grafana panels:

1. **Queue depth** climbs as hey overshoots vLLM throughput
2. ~30-90 seconds in, KEDA's Prometheus query crosses threshold and creates a second vLLM pod
3. The second pod takes 60-180s to load the model and become Ready
4. Once the second replica is serving, queue depth drops
5. After loadgen ends, queue depth stays at 0 for the 5-min cooldown
6. KEDA scales back to 1 replica

What we observed live during plan 04-02 verification on this hardware:
- Peak replica count: **<PEAK_REPLICAS>** (`maxReplicaCount` is 3)
- Time to scale-up (loadgen start → second replica added by KEDA): **~<SCALE_UP_SECONDS>s**
- Time to scale-down after loadgen ended: **~<COOLDOWN_SECONDS>s** (configured `cooldownPeriod: 300s`)

## Part G — Inspect the events

KEDA creates a managed HPA under the hood (named `keda-hpa-vllm-smollm2`):
```bash
kubectl get hpa keda-hpa-vllm-smollm2 -n llm-serving
# This is what KEDA actually drives; the ScaledObject you applied is the higher-level CR.

kubectl describe scaledobject vllm-smollm2 -n llm-serving | tail -20
# Look for: Last Active Time, Conditions: Ready=True, Active=True/False
```

For the contrast: HPA on CPU never fires for the RAG retriever during this loadgen because the RAG retriever isn't being hit. That's the point — HPA on CPU is the right scaler for it; we just have nothing to scale right now. (You can drive the RAG retriever directly with `hey` against `http://rag-retriever.llm-app.svc.cluster.local:8001/search` if you want to see HPA scale it; that's left as an exercise.)

## Common Pitfalls

| Symptom | Root cause | Fix |
|---------|-----------|-----|
| `ScaledObject READY=False`, KEDA logs say "connection refused" | Wrong Prometheus Service name in `serverAddress` | Re-run `scripts/verify-prometheus-svc.sh`, update `serverAddress` in 80-keda-scaledobject-vllm.yaml, `kubectl apply` again |
| `hey` Job pod returns HTTP 404 | Model name in JSON body is `smollm2`, should be `smollm2-135m-finetuned` (Pitfall 3) | Edit 81-loadgen-job-hey.yaml, re-apply |
| `kubectl top nodes` returns "metrics not available" forever | metrics-server didn't get the `--kubelet-insecure-tls` patch | Re-run `scripts/install-metrics-server.sh` (idempotent), or manually `kubectl edit deploy metrics-server -n kube-system` |
| KEDA scales up to `maxReplicaCount: 3` and one pod stays Pending forever | Docker Desktop hit memory ceiling — third vLLM pod can't schedule | Reduce `maxReplicaCount` to `2` in 80-keda-scaledobject-vllm.yaml, OR give Docker Desktop more memory |
| Grafana dashboard never appears under "Smile Dental — Autoscaling" | The CM label `grafana_dashboard: "1"` isn't set | `kubectl get cm grafana-autoscaling-dashboard -n monitoring -o yaml` and confirm the label is present (string `"1"`, not int `1`) |
| ScaledObject `READY=False` with "scaler error: parsing prometheus metadata: query is required" | YAML indentation issue in `triggers[0].metadata` | Re-apply the manifest from disk; verify the indentation matches what's in the repo |

## Summary

You now have:
- KEDA 2.19 installed in `keda` namespace
- metrics-server installed in `kube-system` with KIND patch
- A ScaledObject that drives vLLM replicas based on the Prometheus metric `vllm:num_requests_waiting` — the saturation signal that actually predicts the need for more capacity
- An HPA on the RAG retriever as a contrast moment (CPU-based HPA is the right tool for stateless web services)
- A Grafana dashboard that shows the scale events live, auto-discovered from a labeled ConfigMap
- Live proof that 3 minutes of `hey` loadgen drove KEDA to scale vLLM up and the cooldown brought it back down

The *contrast* (queue depth for LLMs vs CPU for stateless) is the key idea to take into Day 3 — every LLM-serving deployment you build from now on should ask "what's my saturation signal?" before reaching for HPA-on-CPU by default.
