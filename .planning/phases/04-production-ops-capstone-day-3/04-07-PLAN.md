## Part B — Apply RBAC + PVC + copy the LLM API key Secret

The default `argo` ServiceAccount in the `argo` namespace doesn't have `get` on Secrets. Workflow step pods inherit the workflow's SA, and our eval step needs to read `llm-api-keys` and our commit-tag step needs to read `git-deploy-key`. Hence the custom SA (Pitfall 10):

```bash
kubectl apply -f k8s/100-argo-workflows-rbac.yaml
kubectl apply -f k8s/100-pvc-pipeline-workspace.yaml
```

Copy the existing LLM API key Secret from `llm-agent` to `argo` (Workflow step pods can only mount Secrets from their own namespace):

```bash
kubectl get secret llm-api-keys -n llm-agent -o yaml \
  | sed 's/namespace: llm-agent/namespace: argo/' \
  | kubectl apply -f -
```

## Part D — Inspect the WorkflowTemplate (the headline)

Open `k8s/101-workflowtemplate-llm-pipeline.yaml` and look for these things:

1. **The DAG** — 6 named tasks under `spec.templates[0].dag.tasks`:

```yaml
tasks:
- { name: data-gen, template: step-noop, ... }
- { name: train,    template: step-noop, dependencies: [data-gen] }
- { name: merge,    template: step-noop, dependencies: [train] }
- { name: package,  template: step-noop, dependencies: [merge] }
- { name: eval,     template: step-eval,    dependencies: [package] }
- { name: commit-tag, template: step-commit-tag, dependencies: [eval],
    when: "{{tasks.eval.outputs.parameters.pass}} == true" }
```

2. **D-11 short-circuit** — data-gen / train / merge / package use the `step-noop` template (just an `echo`). Re-running real fine-tuning every demo would burn 10-15 min of CPU. The lab teaches the **gate mechanic**, not the training. The "if you have time" extension at the end of the page swaps these in for real.

3. **The eval step writes `pass=true|false` as an output parameter:**

```yaml
- name: step-eval
  container:
    image: kind-registry:5001/smile-dental-deepeval:v1.0.0
    env:
    - { name: VLLM_URL, value: "http://vllm-smollm2.llm-serving.svc.cluster.local:8000" }
    - { name: VLLM_MODEL, value: "smollm2-135m-finetuned" }
    - { name: THRESHOLD, value: "0.7" }
    - name: GROQ_API_KEY
      valueFrom: { secretKeyRef: { name: llm-api-keys, key: groq-api-key } }
  outputs:
    parameters:
    - name: pass
      valueFrom: { path: /tmp/eval-pass.txt }
```

4. **The commit-tag step is gated** — `when: "{{tasks.eval.outputs.parameters.pass}} == true"` means commit-tag is SKIPPED when eval wrote `false`.

5. **The shared PVC** — `volumes[0].persistentVolumeClaim.claimName: pipeline-workspace`, mounted at `/workspace` in every step. No S3 (Pitfall 5).

Apply:

## Common Pitfalls

| Symptom | Root cause | Fix |
|---------|-----------|-----|
| `step-eval` pod stuck in `CreateContainerConfigError` | argo-workflow ServiceAccount can't read `llm-api-keys` Secret (Pitfall 10) | Re-apply `100-argo-workflows-rbac.yaml`; verify with `kubectl auth can-i get secret/llm-api-keys -n argo --as=system:serviceaccount:argo:argo-workflow` |
| `step-eval` pod ImagePullBackOff | Forgot `kind load docker-image` (Pitfall 4) | `kind load docker-image kind-registry:5001/smile-dental-deepeval:v1.0.0 --name llmops-kind` |
| `step-commit-tag` fails with "Permission denied (publickey)" on `git push` | SSH deploy key Secret not mounted, or GitHub deploy key doesn't have "Allow write access" checked | Re-do Lab 11 Part E; ensure the GitHub deploy-key has the write-access checkbox on |
| Workflow stays Pending forever | PVC `pipeline-workspace` is unbound (no PV) | `kubectl describe pvc pipeline-workspace -n argo` — KIND default storageClass should work; if not, `kubectl get sc` and ensure one is `(default)` |
| Eval step returns 429 from Groq | Eval set has > ~25 items at default 2s sleep — saturating 30 RPM | Increase `SLEEP_BETWEEN_CASES` env in `step-eval` from `2.0` to `3.0`+ |
| `step-commit-tag` ran but `git log` shows no new commit | The git-commit-step's `cd /repo && git diff` showed no actual changes (the annotation already had the same value) | Re-run with a new Workflow (the bumped annotation includes a UTC timestamp — should always differ) |
| `when:` gate evaluated false even though eval passed | The eval step wrote something other than `true` (e.g. `True\n` with capital T) | Inspect: `kubectl get workflow <name> -n argo -o jsonpath='{.status.nodes.<eval-node-id>.outputs.parameters[0].value}'` — if it's not the literal string `true`, fix `run_eval.py`'s output write |
| Argo Workflows UI shows nodes but they have no status icons | UI's GraphQL query fails on auth | Restart `argo-workflows-server` pod: `kubectl rollout restart deploy/argo-workflows-server -n argo` |
