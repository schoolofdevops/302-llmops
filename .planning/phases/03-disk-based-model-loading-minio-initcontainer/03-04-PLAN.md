    H1: # Lab 06: Disk-Based Model Loading with MinIO + initContainer

    Duration line: **Day 2 | Duration: ~60 minutes**

    SECTION: ## Learning Objectives (4 bullets)
      - Install MinIO as an in-cluster S3-compatible object store
      - Upload the fine-tuned Smile Dental model from the training artifact to MinIO
      - Deploy vLLM using the initContainer download pattern (Pattern B) with emptyDir, sentinel file, and sha256 verification
      - Understand when to use OCI ImageVolume (Pattern A) vs disk-based loading (Pattern B)

    SECTION: ## Why Disk-Based Loading?
      Brief conceptual framing paragraph (3-4 sentences):
        Lab 03 packaged the model as an OCI image (Pattern A). Pattern A is optimal when the model
        is small (≤2 GB), stable (rare updates), and a container registry is already part of the
        deployment pipeline. For larger models (>2 GB) or frequent retraining cycles, copying a new
        OCI layer for each model version is slow and registry-expensive. Pattern B — disk-based
        loading via MinIO + initContainer — downloads the model from an S3-compatible object store at
        pod startup. The trade-off is explicit: every pod restart re-downloads the model, but model
        updates are as simple as overwriting an S3 object.

    SECTION: ## Part 1: Install MinIO Object Store
      Explain what MinIO is (1-2 sentences: S3-compatible, standalone, course-grade).
      Note: "These credentials (minio / minio123) are hardcoded for the course. Never use
      plaintext credentials in production — use Kubernetes Secrets or external secret managers."

      Commands (using Tabs for macOS/Linux and Windows):
        1. Add Helm repo:
           helm repo add minio-official https://charts.min.io/
           helm repo update minio-official

        2. Create namespace:
           kubectl create namespace minio

        3. Install MinIO chart 5.4.0 using the values file:
           helm install minio minio-official/minio \
             --namespace minio \
             -f course-code/labs/lab-06/solution/k8s/10-minio-values.yaml

        4. Wait for rollout:
           kubectl rollout status deployment/minio -n minio

        5. Verify S3 API (NodePort 30900):
           curl -s http://localhost:30900/minio/health/live
           (Expects: HTTP 200)

        6. Open MinIO Console: http://localhost:30901 (login: minio / minio123)

      Expected state box: 1 pod running in minio namespace; S3 API accessible on localhost:30900.

    SECTION: ## Part 2: Upload the Model to MinIO
      Explain: the model-uploader Job mounts the same /mnt/project directory all KIND nodes share
      via Docker bind-mount and uses `mc` (MinIO Client) to copy the merged model artifact into

    SECTION: ## Part 3: Deploy vLLM with initContainer Download (Pattern B)
      Explain the initContainer pattern:
        The vllm-smollm2-disk Deployment has two containers in the pod:
        1. An initContainer (model-download) that runs first, downloads the model from MinIO into
           a shared emptyDir volume at /model, verifies sha256, then writes a sentinel file /model/READY.
        2. The main vLLM container waits for /model/READY before starting the inference server.
        The emptyDir volume exists only for the lifetime of this pod.

      Memory budget note (callout/admonition):
        ":::warning Memory Budget — Scale Pattern A First
        Running both vLLM Pattern A and Pattern B simultaneously requires ~8 GiB for vLLM pods
        alone, exceeding the 16 GB laptop budget. Scale Pattern A to 0 replicas first:
        kubectl scale deployment vllm-smollm2 -n llm-serving --replicas=0
        :::"

      Commands:
        1. Scale Pattern A to 0 (memory budget):
           kubectl scale deployment vllm-smollm2 -n llm-serving --replicas=0

        2. Apply Service and Deployment:
           kubectl apply -f course-code/labs/lab-06/solution/k8s/30-svc-vllm-disk.yaml
           kubectl apply -f course-code/labs/lab-06/solution/k8s/30-deploy-vllm-disk.yaml

        3. Watch initContainer download (takes ~30-60 seconds):
           kubectl logs -n llm-serving -l app=vllm-disk -c model-download --follow
           Expected: "Downloading model...", "sha256 verified.", "Model download complete."

        4. Watch main container startup (~2-3 minutes after initContainer completes):
           kubectl logs -n llm-serving -l app=vllm-disk -c vllm --follow

        5. Wait for readiness:
           kubectl rollout status deployment/vllm-smollm2-disk -n llm-serving --timeout=900s

        6. Test Pattern B inference:
           curl -s http://localhost:30203/health

        7. Send a chat request to Pattern B:

             -d '{"model":"smollm2-135m-finetuned","messages":[{"role":"user","content":"What dental services do you offer?"}],"max_tokens":50}' | python3 -m json.tool

    SECTION: ## Part 4: Observe the Re-Download Trade-Off
      Framing: "emptyDir is tied to the Pod lifecycle, not the node. When a pod is deleted,
      recreated, or evicted, the emptyDir is wiped and the model must be downloaded again.
      This is the deliberate trade-off of Pattern B."

      Commands:
        1. Note the current pod name:
           kubectl get pods -n llm-serving -l app=vllm-disk

        2. Delete the pod:
           POD=$(kubectl get pods -n llm-serving -l app=vllm-disk -o name | head -1)
           kubectl delete $POD -n llm-serving

        3. Watch the new pod's initContainer download:
           kubectl logs -n llm-serving -l app=vllm-disk -c model-download --follow
           Expected: "Downloading model..." appears again in the new pod — the 517 MB model
           re-downloads in ~30-60 seconds on the local KIND network.

      Teaching point paragraph:
        "In production with a PVC, the model would persist across pod restarts and re-downloads
        would be avoided. Phase 06 (Argo Workflows) uses a PVC-backed MinIO for the training
        pipeline artifact store — that's where you'd use persistent model storage. For this lab,
        the emptyDir re-download makes the trade-off visceral and concrete."

    SECTION: ## Part 5: Pattern A vs Pattern B — Decision Guide
      (This section delivers PACKAGE-03)

      Opening sentence: "Now that you've deployed both patterns, here's when to use each:"

      Include the full comparison table from RESEARCH.md Lab 06 Structure section (6 rows:
      Model size, Update cadence, Registry dependency, Cold-start behavior, Credential management,
      Production choice).

      Decision tree (bullet form):
        Choose Pattern A (OCI ImageVolume) when:
          - Model is ≤ 2 GB (single OCI layer, fits the ImageVolume feature)
          - Model versions are immutable and change infrequently (weekly or less)
          - You already have a container registry in your deployment pipeline
          - Fast cold-start is critical (image layers are node-cached after first pull)

        Choose Pattern B (MinIO + initContainer) when:
          - Model is > 2 GB (exceeds practical OCI layer size for standard registries)
          - Model is retrained frequently (daily or more) — uploading to S3 is instant
          - You already operate an S3-compatible object store (MinIO, AWS S3, GCS)
          - Model updates should not require a CI/CD pipeline rebuild

      Closing note: "Neither pattern is universally superior. The Smile Dental SmolLM2-135M model
      is only 517 MB, making Pattern A optimal in terms of cold-start. We used Pattern B in this
      lab to learn the initContainer pattern, which scales to 7B, 13B, and larger models where
      OCI layers become impractical."

      3-4 bullet recap of what was built.
