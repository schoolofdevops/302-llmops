## Summary

Phase 02.1 is a two-part surgical edit of lab guide markdown files. It touches no source code, no Dockerfiles, no K8s manifests, and no course-code/ files. All changes are confined to `course-content/docs/labs/` markdown files.

**Change 1 вЂ” Flatten workspace paths:** Labs 01-04 currently tell students to copy solution files into `llmops-project/lab-01/`, `llmops-project/lab-02/`, etc. sub-directories, and then `cd llmops-project/lab-01` before running commands. The new model is a single flat workspace: students copy directly into `llmops-project/` and run commands from there. This is already how the K8s layer works вЂ” K8s manifests use `/mnt/project/datasets/train/...` (not `/mnt/project/lab-01/datasets/train/...`) because the KIND volume mounts `./llmops-project` to `/mnt/project`. The per-lab sub-directory pattern was only in the lab guide instructions, not in any code or manifests.

**Change 2 вЂ” Switch pip to uv:** Only two labs have student-facing `pip install` commands (lab-01 and lab-02). Two additional narrative references to `pip install` appear in lab-02's explanation of the initContainer. The K8s initContainer YAML files (`course-code/labs/lab-01/solution/k8s/`) use `pip install` inside their pod `args` вЂ” those are explicitly out of scope and must not change.

**Primary recommendation:** Edit only the 3 lab guides that contain `llmops-project/lab-XX/` paths (lab-01, lab-02, lab-03) and the 2 lab guides that contain student-facing `pip install` (lab-01, lab-02). Apply mechanically: path substitution, then pip-to-uv replacement. Verify that no K8s manifests or Dockerfiles are touched.

### Why This is Already Correct at the K8s Layer

The KIND cluster config mounts `./llmops-project` (the repo's `llmops-project/` directory) to `/mnt/project` on every node:

This means the K8s layer already assumes `datasets/` is directly under `llmops-project/`, not under `llmops-project/lab-01/`. **The per-lab sub-directory pattern in lab guides was creating an inconsistency** вЂ” students following Lab 01 instructions would put files at `llmops-project/lab-01/datasets/` but the K8s Job would look for them at `/mnt/project/datasets/` (which maps to `llmops-project/datasets/`). ## uv Command Reference

**What uv is:** A Python package manager written in Rust by Astral. Drop-in replacement for `pip` with 10-100x faster installs. Installed standalone (does not require pip or Python to install itself).

**Installation (student-facing, added to lab-00 or prerequisites section):**

```bash
# macOS / Linux
curl -LsSf https://astral.sh/uv/install.sh | sh
```

```powershell
# Windows (PowerShell)
powershell -ExecutionPolicy ByPass -c "irm https://astral.sh/uv/install.ps1 | iex"
```

**Student-facing replacement commands:**

| Old (pip) | New (uv) |
|-----------|----------|
| `pip install -r tools/requirements.txt` | `uv pip install -r tools/requirements.txt` |
| `pip install -r rag/requirements.txt` | `uv pip install -r rag/requirements.txt` |
| `pip install some-package` | `uv pip install some-package` |

**Important nuance вЂ” virtual environment behavior:**

`uv pip install` by default requires a virtual environment to be active (unlike `pip`). In a course context where students likely don't manage a venv, there are two options:

1. **`uv pip install --system -r requirements.txt`** вЂ” installs into the system Python. Simplest for students.
2. **`UV_SYSTEM_PYTHON=1 uv pip install -r requirements.txt`** вЂ” same effect via env var.
3. **Activate a venv first** вЂ” more correct but adds complexity for a one-day workshop.

**Recommendation:** Use `uv pip install --system -r requirements.txt` in lab-01 and lab-02 student commands. This is the cleanest single command that works without requiring students to set up a venv. Add a brief note in lab-00 prerequisites explaining `--system` means "use the system Python installation."

**Speed advantage worth mentioning to students:** uv installs `sentence-transformers==5.4.1` (which pulls in PyTorch CPU) in ~30 seconds vs ~3-5 minutes with pip. This is directly relevant to the lab-01 `pip install -r tools/requirements.txt` step which currently has no timing note.

**NOT uv's scope for this phase:** uv project management (`uv init`, `uv add`, `uv.lock`) is out of scope. Only `uv pip install` replaces `pip install` in student commands.

## Common Pitfalls

### Pitfall 1: Accidentally changing K8s initContainer pip commands

**What goes wrong:** Editor searches for `pip install` and replaces ALL occurrences, including inside K8s manifest YAML files embedded in lab guides or in `course-code/`.

**Why it happens:** The lab-02 guide has both a student-facing `pip install -r rag/requirements.txt` AND an explanatory paragraph about the initContainer that mentions `pip install`. The initContainer YAML itself (in `course-code/labs/lab-01/solution/k8s/10-retriever-deployment.yaml`) uses `pip install` inside pod `args`.

**How to avoid:** Change only:
1. Standalone `pip install` lines that students type at their terminal
2. NOT lines inside YAML code blocks showing K8s manifest content
3. NOT files under `course-code/`

**Warning signs:** If any `course-code/` files appear in a diff for this phase, stop вЂ” that's out of scope.

### Pitfall 2: Breaking the cross-lab data dependency

**What goes wrong:** Flattening lab-01's workspace changes where `datasets/train/dental_chat.jsonl` lives. Lab-03 checks for this file. If the check command is not also updated, students see a false "missing file" error in Lab 03.

**How to avoid:** The path change in lab-03 is already identified above: `ls llmops-project/lab-01/datasets/train/...` в†’ `ls llmops-project/datasets/train/...`.

**Warning signs:** After editing, grep for any remaining `lab-01/datasets` references across all lab guides вЂ” there should be zero.

### Pitfall 3: Windows path separators in xcopy commands

**What goes wrong:** When converting `xcopy /E /I course-code\labs\lab-01\solution\* llmops-project\lab-01\` to the flat version, it's tempting to write `llmops-project\datasets` вЂ” but the `xcopy` wildcard `\*` already copies all subdirectories. The target is just `llmops-project\`.

**How to avoid:** The xcopy target in Windows commands should be `llmops-project\` (the top-level workspace), matching the macOS `cp -r ... llmops-project/`. Do not try to map individual subdirectory targets.

### Pitfall 4: uv --system flag on lab-02's local build-index step

**What goes wrong:** Lab-02, Step 2 is an optional local run of `build_index.py` for understanding. It runs `pip install -r rag/requirements.txt` before `python rag/build_index.py`. The `--system` flag must be included in the replacement here too.

**How to avoid:** Search specifically for `pip install -r rag/requirements.txt` (not just `pip install`) to ensure this specific occurrence is found and updated.

### Pitfall 5: mkdir -p command creates subdirectory in the wrong location

**What goes wrong:** Old command `mkdir -p llmops-project/lab-01` вЂ” students run this from repo root. New flat structure needs the target directories to exist but `cp -r ... llmops-project/` will create them automatically from the solution directory structure. The `mkdir -p` command may become redundant or needs updating.

**How to avoid:** In the flat model, `cp -r course-code/labs/lab-01/solution/* llmops-project/` is sufficient вЂ” it copies datasets/, rag/, tools/, k8s/ directly into llmops-project/. The `mkdir -p` line can be replaced with a note explaining what the cp creates, OR simplified to just `mkdir -p llmops-project`.

## Open Questions

1. **Does lab-01's `mkdir -p llmops-project/lab-01` need to be replaced with anything, or is it redundant?**
   - What we know: `cp -r course-code/labs/lab-01/solution/* llmops-project/` will create subdirectories automatically
   - What's unclear: Whether `llmops-project/` itself needs to be created first, or whether it already exists from the KIND bootstrap step
   - Recommendation: Replace with `mkdir -p llmops-project` (ensure parent exists) вЂ” the `cp -r` handles subdirectories. Alternatively, the bootstrap script could create the directory and students just run the cp.

2. **Should uv installation be added to lab-00 (cluster setup) or assumed as a prerequisite in the course preflight script?**
   - What we know: lab-00 is currently a stub ("Full lab instructions coming in Phase 2"). The preflight script (`scripts/preflight-check.sh`) already checks Docker, KIND, kubectl, Helm.
   - What's unclear: Whether Phase 02.1 should add uv to the preflight check or leave that for Phase 3 (which will author lab-00 fully)
   - Recommendation: Add a simple uv installation note at the top of lab-01 (the first lab where students use it), mentioning the `astral.sh/uv` installer. Do not modify the preflight script in this phase вЂ” that scope belongs to lab-00 authoring.

3. **Windows: does `uv pip install --system` work correctly on Windows with Docker Desktop Python?**
   - What we know: uv supports Windows via PowerShell installer (`irm https://astral.sh/uv/install.ps1 | iex`). `uv pip install --system` works on macOS.
   - What's unclear: Whether Docker Desktop's Python environment on Windows requires `--break-system-packages` in addition to `--system`
   - Recommendation: Document the standard `--system` flag and add a note that Windows users who see an "EXTERNALLY-MANAGED" error should add `--break-system-packages`. This is a LOW confidence risk вЂ” most student machines likely don't have EXTERNALLY-MANAGED Python on Windows.
